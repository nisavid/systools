# mlxctl deployment contract v1

This reference defines the supported boundary between mlxctl and a deployment
owner such as chezmoi. The deployment owner installs the package and may apply
the inactive LaunchAgent. mlxctl owns desired state, Runtime Installations,
models, services, client configuration, operational state, and lifecycle.

## Installed entry points

| Entry point | Contract |
| --- | --- |
| `mlxctl` | Human and machine CLI. No arguments open the TUI. |
| `mlxd` | Foreground per-user Supervisor used by launchd. |
| `python -m mlxctl.entrypoints daemon` | Private stable target used by mlxctl's generated LaunchAgent. |

Install from `tools/mlxctl/` with `uv tool install`. Do not install separate
MLX runtimes globally for mlxctl: `mlxctl runtime install` owns isolated exact
Runtime Installations.

## LaunchAgent

- Label: `io.nisavid.mlxd`
- Path: `~/Library/LaunchAgents/io.nisavid.mlxd.plist`
- `RunAtLoad`: `false`
- `KeepAlive`: `false`
- `Umask`: `0077`
- stdout and stderr: `~/Library/Logs/mlxctl/supervisor.log`

mlxctl generates a literal executable path and the private module target in
`ProgramArguments`. The LaunchAgent is registered but inactive. A mutation
that needs the Supervisor registers or kickstarts it and waits for the private
control socket. Read-only operations never activate it.

`mlxctl supervisor stop` drains all Service Runs, stops the Gateway, replies to
the caller, closes the control service, and lets `mlxd` exit. `launchd` does not
restart it.

## Owned paths

| Purpose | Default | Environment override |
| --- | --- | --- |
| Desired state | `~/.config/mlxctl/config.toml` | `MLXCTL_CONFIG_DIR` |
| SQLite state and socket | `~/.local/state/mlxctl/` | `MLXCTL_STATE_DIR` |
| Codex model catalog and ownership | `~/.local/state/mlxctl/clients/` | `MLXCTL_STATE_DIR` |
| Runtime Installations | `~/.local/share/mlxctl/runtimes/` | `MLXCTL_DATA_DIR` |
| Logs | `~/Library/Logs/mlxctl/` | `MLXCTL_LOG_DIR` |
| `uv` executable | Discovered from known absolute paths | `MLXCTL_UV_EXECUTABLE` |

Directories are user-owned, non-symlink directories with mode `0700`.
Supervisor and service logs are user-owned regular files with mode `0600`.

The Hugging Face cache remains shared and is not product-owned. mlxctl manages
its bytes through model-cache operations and reference checks.

## Ownership boundary

The deployment owner may:

- install or update the mlxctl package from an immutable systools revision;
- render and register the inactive LaunchAgent generated by the supported
  production composition;
- invoke `mlxctl setup` or other public CLI operations after installation;
- verify package entry points and inactive launchd registration.

The deployment owner must not author or mutate:

- `config.toml` service, model, runtime, Gateway, or client tables;
- isolated Runtime Installation directories;
- model cache contents;
- Codex provider or model-catalog settings and Hindsight settings that mlxctl owns;
- SQLite operational state or the control socket;
- live Service Runs or Gateway state outside the mlxctl interface.

This keeps `chezmoi apply` idempotent and prevents two systems from competing
over the same desired state.

## Control protocol

`mlxctl` and `mlxd` use a private, versioned, length-prefixed JSON protocol on
`~/.local/state/mlxctl/mlxd.sock`. Each connection negotiates v1, carries one
correlated operation, streams bounded progress frames, and returns one result
or stable error. Frames are size-bounded. The socket is loopback-equivalent
local IPC and is never exposed on a network interface.

Physical runtime and model operations receive durable operation identities.
Public v1 supports listing and inspecting those operations. It does not claim
resume or cancellation semantics that an owner cannot guarantee.

## Gateway and runtime processes

The Gateway binds to a literal loopback address, defaulting to
`127.0.0.1:8766`, and exposes OpenAI-compatible routes below `/v1`. Each
Inference Service process binds to a private dynamic loopback port. The public
request `model` field selects the stable service route.

Several services may run concurrently. The Gateway does not start a stopped
service in response to traffic. Runtime processes are launched from exact
owned installations with validated argv, capabilities, cached model identity,
and revision/runtime-scoped trust grants.

## Deployment validation

A deployment change is complete only after these checks pass on a target Mac:

```sh
mlxctl --help
mlxd --help
mlxctl status
mlxctl runtime available
mlxctl setup --help
mlxctl supervisor start
mlxctl supervisor status
mlxctl supervisor stop
```

After configuring Codex, `mlxctl client inspect codex` must report a healthy
owned catalog, and `codex debug models` must list the selected route with the
same context cap without a fallback-metadata warning.

On the target Mac with the service running, send one bounded Codex request and
confirm its output or logs contain no fallback-metadata warning. If the
installed Codex exposes custom catalogs through app-server `model/list`, verify
the same route and context there as well; versions that return only the bundled
catalog use `codex debug models` as the custom-catalog acceptance surface.

When a model service is part of the deployment, also verify its exact Runtime
Installation, Model Revision, launch argv, Gateway route, `/v1/models`, one
bounded completion request, logs, metrics, and clean service stop. A status
read after stop must not reactivate `mlxd`.
